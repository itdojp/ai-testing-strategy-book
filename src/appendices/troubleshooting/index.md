# AI生成コードとテスト失敗の安全な切り分け

AI生成コード、生成テスト、eval/metric、CI/MLOpsで失敗を見つけた場合は、速度よりも影響の封じ込めを優先する。この付録は、変更を安全に最小化し、判断根拠を残すための読者向け手順である。

## 使い方

1. 失敗の症状に対応する入口から開始する。
2. すべての症状で、次の安全確認から記録までの順序を守る。
3. 本番への変更、秘密情報を含むログの共有、権限拡大、データ破壊につながる操作は、この付録だけで判断せず、責任者へエスカレーションする。

## 症状別の入口

- **AI生成コードが期待どおりに動かない**: [第1章の生成コードの信頼性](../../chapters/chapter-01-ai-driven-development/#figure-ai-human-code-comparison) と [第5章の検証技法](../../chapters/chapter-05-ai-code-verification/) を確認する。
- **生成テストが失敗・flaky・無意味に通過する**: [第2章のテストレベル](../../chapters/chapter-02-software-testing-fundamentals/#figure-ai-testing-pyramid) と [第6章の生成テスト品質保証](../../chapters/chapter-06-test-automation-ai-cooperation/) を確認する。
- **eval/metric が悪化または比較不能になった**: [第7章の品質評価](../../chapters/chapter-07-quality-metrics-evaluation/#figure-quality-evaluation-framework) を確認する。
- **CI/MLOps の品質ゲートまたは監視で失敗した**: [第6章のCI/CD統合](../../chapters/chapter-06-test-automation-ai-cooperation/#figure-mlops-pipeline) を確認する。

## 安全な切り分けフロー

### 1. 症状を固定する

失敗したコマンド、入力、期待結果、実際の結果、実行時刻、commit、model/runtime/SDK、runner、環境を記録する。ログやpromptに個人情報、認証情報、顧客データ、未公開仕様が含まれる場合は、先にredactionし、承認済みの保管場所だけを使う。

### 2. 安全確認をする

対象環境、権限、データ分類、変更可能な範囲を確認する。本番、共有環境、顧客データ、外部AIサービスへの入力が関わる場合は、read-onlyの観察へ限定する。削除・再学習・再デプロイ・権限変更は承認なしに行わない。

### 3. 再現可能な最小条件を作る

隔離した開発環境で、固定したseed、fixture、依存関係、入力を用いて最小の失敗例を作る。生成AIの出力は非決定的になり得るため、prompt、model、temperature、tool、dataset版を固定し、同じ条件で再実行する。再現できない場合は推測で修正しない。

### 4. 原因仮説を一つずつ分ける

生成コードなら要件誤読・境界条件・依存関係・セキュリティ観点、生成テストならoracle・fixture・非決定性・期待値、eval/metricなら評価条件・dataset・しきい値・集計、CI/MLOpsなら設定・artifact・資格情報・環境差・監視条件に分ける。複数の変更を同時に加えない。

### 5. 最小の安全な検証を行う

read-onlyの検査、単体テスト、固定fixtureの小さなeval、dry-run、限定的なCI jobを優先する。結果は「仮説を支持した」「反証した」「判断不能」のいずれかで記録する。通過しただけでは、品質や安全性が保証されたと結論付けない。

### 6. 停止・エスカレーションする

次の場合は変更を停止し、担当の開発責任者、QA、SRE/Platform、Security/Privacy、またはモデル運用責任者へエスカレーションする。

- 本番影響、データ破壊、セキュリティ脆弱性、個人情報・秘密情報の露出が疑われる。
- 同一条件で再現できない、または失敗範囲を限定できない。
- 期待値、評価基準、承認者が不明である。
- 権限昇格、外部サービスへの機密入力、モデル再学習・再デプロイが必要になる。

### 7. 判断と証跡を残す

症状、影響範囲、安全確認、再現手順、固定した条件、仮説、最小検証の結果、停止またはエスカレーションの判断、次の責任者を記録する。機密情報は含めず、参照先とredactionの有無を明記する。

## 最小記録テンプレート

| 項目 | 記録する内容 |
| --- | --- |
| 症状 | 失敗した操作、期待結果、実際の結果 |
| 影響 | 開発・CI・staging・本番の別、対象利用者・データ |
| 条件 | commit、入力、seed、model/runtime/SDK、runner、時刻 |
| 安全確認 | 権限、データ分類、redaction、承認の状態 |
| 仮説と検証 | 一度に一つの仮説、最小検証、結果 |
| 判断 | 継続・停止・エスカレーション、担当者、次の確認時刻 |

> この付録は原因の断定や本番操作の手順ではない。安全な観察、最小再現、責任分界、証跡化を支援するための導線である。
